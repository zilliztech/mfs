# s3 connector — search & browse

## URI tree

```
s3://<alias>/
└── <key-prefix>...                  ← mirrors the object keys under the configured prefix
```

If `prefix = "docs/2026/"` in the toml, the URI tree starts there.

## Record / object shape

Each S3 object is one file-like MFS object. Conversion routes by media
type:
- PDFs → markdown via markitdown → `chunk_body`
- markdown / text / code → chunked directly
- images → VLM description (if summary enabled) → `vlm_description`

## Chunk kinds

- **`chunk_body`** per converted-text file (chonkie chunker)
- **`vlm_description`** per image (only if VLM enabled)
- **`directory_summary`** per dir (only if summary enabled,
  bottom-up roll-up)

## Locator

```bash
mfs cat s3://<alias>/docs/2026/rfc-001.pdf --range 100:200
```

Locator for text chunks: `{"lines": [s, e]}`.

## Search strategy

| Intent | Use |
|---|---|
| Find docs about X | `mfs search "X" s3://<alias>` |
| Find images by description | (only with VLM enabled) `mfs search "X" s3://<alias> --kind vlm_description` |
| Get raw bytes | `mfs export s3://<alias>/<key> /local/copy.pdf` (uses S3 directly, bypasses chunk store) |

## Pitfalls

- **`prefix` is exact**: if `prefix = "docs"` (no slash) matches
  `docs-archive/*` too. Use `"docs/"` for strict directory semantics.
- **Versioned buckets**: only latest version per key is indexed.
- **Very large PDFs**: markitdown conversion is heavy; a 500-page PDF
  takes a while. The converted_md artifact is cached in the artifact
  cache so subsequent re-syncs reuse it.
- **Glacier-tier objects**: not indexable (the connector can't read
  them without first restoring). They appear in `mfs ls` but
  `search_status: unavailable`.
